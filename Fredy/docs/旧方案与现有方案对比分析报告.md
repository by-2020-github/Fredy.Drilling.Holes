# 旧方案与现有方案对比分析报告

## 范围

本报告对比以下旧方案与现有方案：

- 旧方案
  - `Fredy\Windows\ParamConfig\old\ConfigManager.cs`
  - `Fredy\Windows\ParamConfig\old\ProcessParameters.cs`
  - `Fredy\Windows\ParamConfig\old\ProcessManager.cs`
- 现有方案
  - `Fredy\Services\ConfigService.cs`
  - `Fredy\Windows\ParamConfig\ConfigModels.cs`
  - `Fredy\Windows\ParamConfig\ConfigViewModel.cs`
  - `Common\Models\Recipe.cs`
  - `Common\Tools\RecipeService.cs`
  - `BLL\IMotionService.cs`
  - `BLL\Hardware\Adt8940Controller.cs`
  - `HAL\Motion\MotionAdt8940.cs`

目标：
- 对比旧方案与现有方案的职责划分
- 分析旧方案覆盖的能力在现有方案中是否有遗漏
- 给出迁移与优化建议

---

## 一、总体结论

### 结论摘要
现有方案在**配置持久化、参数分层、运行时下发、配方管理**方面，已经明显优于旧方案。

但从旧方案迁移到现有方案后，仍存在以下几类潜在遗漏或未完全承接的点：

1. **旧 `ProcessParameters` 中部分“工艺过程专属字段”未完整映射到现有模型**
2. **旧 `ProcessManager` 中的回零流程细节，目前仅部分由现有控制器/运动层承接**
3. **旧方案中的流程状态机、异常/急停处理语义，没有完整沉淀为新架构下的统一服务能力**
4. **旧配置是“宽松键值字典”，现有配置是“强类型模型”，优点明显，但迁移时更容易遗漏旧字段**

因此：

- **现有方案方向是正确的，应继续沿用**
- 但建议对旧字段和旧回零/流程逻辑做一次系统性盘点与归档，避免“旧代码里有能力，新系统里没有明确承接点”

---

## 二、旧方案与现有方案的设计差异

## 1. 配置管理方式对比

### 旧方案：`ConfigManager`
特点：
- 单例
- 基于 `Dictionary<string, object>` 存储
- 通过字符串 key 读写配置
- 支持点路径访问嵌套项
- 读取宽松，类型转换大量依赖运行时转换

优点：
- 灵活
- 扩展字段门槛低

缺点：
- 类型不安全
- key 拼写容易出错
- 不利于重构和 IDE 导航
- 很难系统知道“哪些配置真正存在”

### 现有方案：`ConfigService + AppConfig`
特点：
- 强类型模型
- 线程安全访问
- 保存时归档历史配置
- 通过 `ConfigViewModel` 驱动 UI 编辑
- 可与运行时服务联动（如 `IMotionService.ConfigureAxes(...)`）

优点：
- 类型安全
- 易维护
- 可追踪字段来源
- 便于分层（轴参数、工艺参数、相机参数等）

结论：
- **现有配置方案明显优于旧方案**
- 旧 `ConfigManager` 不建议恢复使用
- 如果需要兼容旧配置，只建议做“迁移读取器”，不要继续扩展旧字典方案

---

## 2. 工艺参数管理方式对比

### 旧方案：`ProcessParameters`
特点：
- 单类集中承载全部工艺参数
- 混合了：
  - 工件参数
  - 校针参数
  - 工件基准参数
  - 头道/二道冲孔参数
- 直接读写单独文件 `ProcessParameters.json`

问题：
- 单个模型职责过重
- 参数粒度混杂
- 难区分“全局默认工艺参数”和“配方专属参数”

### 现有方案
当前工艺参数分散在两层：

#### 全局配置层：`AppConfig`
承载：
- `FirstPass`
- `SecondPass`
- `FastMovePos`
- `FastMoveSpeed`
- `SlowMoveDist`
- `SlowMoveSpeed`
- 检测阈值
- 回零模式相关项

#### 配方层：`RecipeProcessParameters`
承载：
- `WorkpieceType`
- `FirstPunchDepth`
- `FirstAlarmDepth`
- `FirstLiftHeight`
- `FirstPeckDepth`
- `FirstPeckSingleDepth`
- `SecondPunchDepth`
- `SecondAlarmDepth`
- `SecondLiftHeight`
- `MinSafeDepth`
- `IsSecondDetectionActive`
- 一道/二道探测列表与偏移阈值

结论：
- **现有方案在“全局默认值”与“配方实例值”之间已有更合理分层**
- 比旧 `ProcessParameters` 更适合长期维护

---

## 3. 过程控制方式对比

### 旧方案：`ProcessManager`
特点：
- 单例
- 自己直接操作旧控制卡对象
- 内置回零流程
- 内置急停处理
- 内置第一道/第二道流程状态枚举
- 带重试、阻塞轮询、日志输出、UI MessageBox

问题：
- UI、业务、硬件逻辑耦合严重
- 阻塞式写法较多
- 与全局静态参数高度耦合
- 难测试、难复用

### 现有方案
现有方案已拆成：
- `IMotionService` / `MotionManager`
- `IMoton` / `MotionAdt8940` / `MotionSimulator`
- `IHardwareController` / `Adt8940Controller`
- `RecipeService`
- `ConfigService`
- 业务流程服务（如 `AutoPunchMachine`）

结论：
- **现有架构方向优于旧 `ProcessManager`**
- 但旧方案里一些“细节能力”还未完全显式迁移

---

## 三、旧方案能力在现有方案中的承接情况

## 1. `ProcessParameters` 字段覆盖情况

### 旧字段中已被现有方案承接的部分

#### 工件参数
- 旧：`WorkpieceType`
- 现：`RecipeProcessParameters.WorkpieceType`
- 结论：**已承接**

#### 头道冲孔参数
- 旧：
  - `FirstPunchDepth`
  - `FirstPunchBrokenNeedleAlarmDepth`
  - `FirstPunchLiftHeight`
  - `FirstPunchPeckDepth`
  - `FirstPunchPeckSingleDepth`
- 现：
  - `FirstPunchDepth`
  - `FirstAlarmDepth`
  - `FirstLiftHeight`
  - `FirstPeckDepth`
  - `FirstPeckSingleDepth`
- 结论：**基本已承接，仅字段命名更规范**

#### 二道冲孔参数
- 旧：
  - `SecondPunchDepth`
  - `SecondPunchBrokenNeedleAlarmDepth`
  - `SecondPunchLiftHeight`
  - `SecondPunchMinSafeDepth`
  - `SecondPunchDetection`
- 现：
  - `SecondPunchDepth`
  - `SecondAlarmDepth`
  - `SecondLiftHeight`
  - `MinSafeDepth`
  - `IsSecondDetectionActive`
- 结论：**已承接**

#### 探测参数
- 旧方案未形成列表化结构，但流程层有探测处理概念
- 现有方案：
  - `AppConfig.DetectionRingItems`
  - `AppConfig.SecondPassDetectionItems`
  - `RecipeProcessParameters.FirstPassDetectionItems`
  - `RecipeProcessParameters.SecondPassDetectionItems`
- 结论：**现有方案更强**

---

### 旧字段中未明显承接或仅部分承接的部分

#### 校针工艺参数
旧：
- `NeedleCalibrationCameraX`
- `NeedleCalibrationCameraY`
- `NeedleCalibrationNeedleX`
- `NeedleCalibrationNeedleY`
- `NeedleCalibrationOffsetX`
- `NeedleCalibrationOffsetY`
- `NeedleCalibrationStatus`

现状：
- 有新的 `NinePointCalibrationViewModel`
- 有相机像素尺寸标定流程
- 但从当前模型看，没有发现与上述旧字段一一对应的持久化模型

结论：
- **疑似存在迁移遗漏**
- 需要确认旧“校针工艺参数”是否已被新的标定流程完全替代
- 若未替代，应明确：
  - 是否仍需要持久化针尖/相机偏移
  - 如果需要，应落到新的强类型配置或配方模型中

#### 工件基准工艺参数
旧：
- `WorkpieceReferenceX`
- `WorkpieceReferenceY`
- `WorkpieceReferenceZ`
- `WorkpieceReferenceTolerance`
- `WorkpieceReferenceAutoCalibrate`

现状：
- 当前配置模型与配方模型中未看到明显一一对应字段

结论：
- **疑似存在迁移遗漏**
- 若现有业务仍有“工件基准定位/自动校准”需求，应单独建立：
  - `WorkpieceReferenceConfig`
  - 或并入配方/定位模块配置

## 2. `ProcessManager` 回零逻辑覆盖情况

### 旧方案具备的能力
从 `old\ProcessManager.cs` 可见，旧回零流程至少包含：

- XY/Z 分轴回零策略不同
- 机械零位开关极性判断
- IO 端口读取
- 连续运动 + 轮询检测
- 超时保护
- 多次重试
- 光栅尺回零模式
- 锁存/IO 模式切换
- Z 轴回零后抬高指定距离
- 紧急停止标志联动
- 回零标志位更新

### 现有方案具备的能力
现有方案已有：
- `IMotionService.HomeX/Y/Z/All`
- `MotionAdt8940.HomeAsync`
- `Config` 中有：
  - `HomeSearchSpeed`
  - `IsIoHome`
  - `IsLatch`
  - `IsGratingHome`
- `MotionAdt8940` 暴露：
  - `SetLockPositionModeAsync`
  - `GetLockStatusAsync`
  - `GetLockPositionAsync`
  - `ClearLockStatusAsync`

### 差距分析
#### 已承接
- 基础回零动作
- 锁存相关底层能力
- 回零模式参数的配置入口

#### 未完全承接
- XY/Z 分支化机械回零流程细节
- IO 极性/端口级回零逻辑在新架构中的专门服务化封装
- 回零超时与重试策略
- Z 轴回零后抬高逻辑
- 急停联动下的完整回零中断与清理

结论：
- **回零相关的“底层能力”已具备一部分，但“流程编排能力”未完全迁移**
- 现有系统若未来要替代旧回零方案，建议单独实现 `HomingService` 或 `HomeSequenceService`

## 五、现有方案可能的遗漏清单

以下内容建议重点确认：

### A. 校针参数是否完全迁移
旧字段：
- `NeedleCalibrationCameraX/Y`
- `NeedleCalibrationNeedleX/Y`
- `NeedleCalibrationOffsetX/Y`
- `NeedleCalibrationStatus`

现状：
- 未看到等价强类型配置承接

建议：
- 确认是否已由新标定方案完全替代
- 若未替代，建立专门配置模型并迁移

### B. 工件基准参数是否仍需要
旧字段：
- `WorkpieceReferenceX/Y/Z`
- `WorkpieceReferenceTolerance`
- `WorkpieceReferenceAutoCalibrate`

现状：
- 当前模型未看到明确承接

建议：
- 确认业务是否还需要
- 若需要，单独建模，不要塞进 `AppConfig` 顶层杂项中

### C. 回零流程策略是否缺少服务化封装
旧能力：
- 机械回零
- 光栅尺回零
- IO 模式 / 锁存模式
- 重试 / 超时 / 抬高 / 急停清理

现状：
- 参数有，部分底层接口有，但流程服务未完整收敛

建议：
- 新增 `HomeSequenceService` 或 `IHomingService`
- 由它读取 `AppConfig` 和 `IMotionService/IIOCard` 编排回零

### D. 工艺默认值与配方覆盖关系需要进一步明确
现状：
- 一部分工艺参数在 `AppConfig`
- 一部分在 `RecipeProcessParameters`
- `RecipeViewModel` 中已有“配方优先、全局兜底”模式

建议：
- 建立明确规则文档：
  - 哪些是“全局默认”
  - 哪些是“配方必填”
  - 哪些允许“配方缺省时回退到全局配置”

## 六、更改建议

## 建议一：保留现有总体架构，不回退旧方案
不建议恢复：
- `ConfigManager`
- `ProcessParameters`
- `ProcessManager`

旧代码可作为：
- 需求对照
- 迁移参考
- 旧逻辑归档

---

## 建议二：对旧字段做“迁移核对表”
建议新增一份迁移表，逐项确认：

- 旧字段名称
- 旧含义
- 现有承接位置
- 状态：已迁移 / 部分迁移 / 未迁移 / 已废弃

这一步能防止遗漏隐性参数。

---

## 建议三：补一个专门的回零流程服务
建议新增：
- `IHomingService`
- `HomingService`

职责：
- 读取回零相关配置
- 根据 X/Y/Z 轴选择策略
- 执行 IO/锁存/光栅尺回零流程
- 实现超时、重试、急停清理、抬高等旧逻辑

这样可把旧 `ProcessManager` 中最关键的流程能力沉淀到新架构里。

---

## 建议四：把“校针/工件基准”从通用配置中单独归类
若业务仍需要，建议新增专门模型，例如：
- `NeedleCalibrationConfig`
- `WorkpieceReferenceConfig`

不要继续塞进一个大而全的 `AppConfig` 杂项区。

---

## 建议五：继续推进“轴参数”和“工艺参数”分离
当前你们已经开始做这件事，这是正确方向。

建议继续坚持：
- 轴级参数 -> `AxisParamConfig`
- 工艺节拍/冲孔逻辑 -> `FirstPass` / `SecondPass` / 探测参数
- 控制器专项功能 -> 单独配置对象

---

## 七、最终判断

### 现有方案是否优于旧方案？
**是。**

### 现有方案是否完全覆盖旧方案？
**还没有完全覆盖。**

### 主要遗漏风险在哪里？
1. 校针参数
2. 工件基准参数
3. 回零流程策略服务化
4. 旧流程状态与异常处理细节

### 下一步最值得优先做什么？
优先级建议：

1. **做旧字段迁移核对表**
2. **确认校针/工件基准参数是否还需要**
3. **设计并补充回零流程服务**
4. **继续清理 UI 中轴参数与工艺参数的边界**

---

## 八、建议结论（可执行版）

建议按以下策略推进：

- 保留旧代码，仅作为参考，不再扩展
- 现有 `ConfigService + AppConfig + RecipeService + MotionService` 继续作为主方案
- 先补“迁移核对表”，确认旧字段去向
- 对确认仍有业务价值但尚未承接的旧字段，分别归类到：
  - 配方参数
  - 全局工艺参数
  - 校准参数
  - 回零专项配置
- 对旧 `ProcessManager` 中的回零流程，单独抽象新服务承接，而不是继续堆在控制器类里
